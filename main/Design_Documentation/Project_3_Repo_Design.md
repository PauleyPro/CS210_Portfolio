# Project Three Repo Mapping

## Purpose

This document maps the repository structure for Project Three in a way that aligns the codebase with the assignment requirements. The goal is to make the repository easy to understand, easy to grade, and easy to build step by step.

---

## Core Design Idea

The repository is organized by **functional responsibility** and **grading requirement**.

* `main.cpp` controls the overall program loop and user interaction flow.
* `.vscode` stores the workspace build and debug configuration used to compile and run the project.
* `Data_Management` contains the frequency-processing logic and backup data file.
* `Include` contains the class declaration used across the program.
* `Menus` is broken down by build stages, requirements, and test planning so each menu option can be understood independently.
* `Test_Logs` stores generated test executables so build output does not clutter the repository root.
* `Design_Documentation` tracks project planning, annotation standards, source citations, and other support documentation.

This structure is intentionally more descriptive than a standard beginner C++ project so that an instructor or grader can quickly identify how each part of the assignment is being handled.

## Root-Level File Explanation

### `.gitignore`

**Purpose**

This file keeps generated executables and other build output from being treated like source-controlled project files.

**Responsibility**

* ignore the main compiled application
* ignore generated test executables
* allow the `Test_Logs` folder structure to stay in the repository through `.gitkeep`

### `grocery_tracker_app`

**Purpose**

This file is the compiled executable for the full Corner Grocer application.

**Responsibility**

* provide the runnable output of the main project build
* combine `main.cpp`, data-management logic, and menu stages into one executable

### `main.cpp`

**Purpose**

This file serves as the control center for the program.

**Responsibility**

* start the program
* create the `GroceryTracker` object
* load grocery purchase data
* create the backup frequency file
* display the menu repeatedly
* route the user to the correct menu action
* end the program when option 4 is selected

**Why It Belongs at the Root**

Keeping `main.cpp` at the top level makes it easy to locate the primary entry point of the program. This is especially useful for grading because it allows the reviewer to quickly identify where execution begins.

## .vscode Folder

### Purpose

This folder stores the workspace-specific build and debug configuration used in VS Code.

### Files

#### `.vscode/tasks.json`

This file defines build tasks for the main application and the test executables.

**Responsibilities**

* compile the full application into `grocery_tracker_app`
* compile test executables into `Test_Logs`
* keep build steps repeatable inside the editor

#### `.vscode/launch.json`

This file defines the debugger launch configuration for the main application.

**Responsibilities**

* launch the compiled app in the debugger
* call the matching pre-launch build task
* keep debug setup consistent for local testing

### Why This Folder Exists

This folder makes the project easier to run and debug inside VS Code without having to retype the full compile command each time.

## Data_Management Folder

### Purpose

This folder contains the logic responsible for handling grocery item data.

### Files

#### `Data_Management/GroceryTracker.cpp`

This file contains the implementation of the grocery tracking class.

**Responsibilities**

* open and read the input file
* count item frequencies
* store frequency values in the map
* search for a specific item
* print all item frequencies
* print the histogram
* create the `frequency.dat` backup file

#### `Data_Management/CS210_Project_Three_Input_File.txt`

This file stores the source grocery purchase entries that the program reads when building item frequencies.

**Responsibilities**

* provide the raw item names used as input data
* support repeatable testing of search, listing, and histogram output

#### `Data_Management/frequency.dat`

This file is the required backup data output.

**Responsibilities**

* store all grocery item names and counts
* provide a backup of processed frequency data
* reflect the data gathered from the original input file

#### `Data_Management/README.md`

This file explains the purpose of the data management section.

**It should describe**

* what data is handled here
* how the program reads and writes files
* how `GroceryTracker.cpp` works with `GroceryTracker.h`
* why `frequency.dat` is created automatically

### Why This Folder Exists

This folder isolates the data-processing portion of the assignment so it can be reviewed separately from the menu logic. That improves clarity, maintainability, and grading readability.

## Include Folder

### Purpose

This folder contains the class declaration used by the program.

### Files

#### `Include/GroceryTracker.h`

This file defines the grocery tracking class.

**Responsibilities**

* declare private data members
* declare public member functions
* provide the interface used by `main.cpp`
* separate class design from class implementation

#### `Include/Menu_UI.h`, `Menu_Handling.h`, `Menu_Search.h`, `Menu_Listed.h`, `Menu_Histogram.h`, `Menu_Exit.h`

These files define the shared function declarations for each menu stage.

**Responsibilities**

* publish stage function declarations for cross-file use
* reduce manual forward declarations in source files
* keep stage interfaces separated from stage implementations

### Why This Folder Exists

Using a dedicated include folder supports standard C++ project organization and makes it clear where the class definition is stored.

## Recommended Usage Notes

To keep the project practical, only the actual code files should be required for compilation:

* `main.cpp`
* `Include/GroceryTracker.h`
* `Data_Management/GroceryTracker.cpp`
* `Menus/Stages/*.cpp`

The `.vscode` folder supports editor-based build and debug workflows, while `Test_Logs` and `grocery_tracker_app` are generated output locations rather than source-design areas.

This means the project stays easy to build while still giving the repository a highly organized instructional structure.
